# セキュリティ設計ガイド（AI 向け）

このドキュメントは、アプリケーション実装時に **セキュリティリスクの高いコードを書かないための最低限のルール** をまとめたものです。  
AI がコード生成・改修を行う際は、ここに書かれた方針に従ってください。

---

## 1. 認証・認可

### 1.1 パスワード取り扱い

- **パスワードの平文保存は禁止。**
  - DB にはハッシュ化した値のみ保存すること。
  - ハッシュには、単純な `sha256` などを直接使わず、`bcrypt` / `scrypt` / `Argon2` など、用途に適した KDF を利用する。
- パスワードやシークレットをログに出力してはいけない。
- パスワードリセット機能を実装する場合は、**一時トークン**を発行し、トークンは短い有効期限・1回限り使用可能にする。

### 1.2 セッション / トークン

- JWT やセッショントークンは **署名付き** にし、推測困難な値にする。
- トークンの有効期限（`exp`）を必ず設定し、長期間有効なトークンは発行しない。
- ブラウザ Cookie でセッションを扱う場合:
  - `HttpOnly` フラグを付け、JavaScript から参照できないようにする（XSS 被害時の影響を軽減）。
  - HTTPS 前提の環境では `Secure` フラグを付ける。

### 1.3 認可（権限チェック）

- 認証済みであることと、**その操作を行う権限があるか** は分けてチェックする。
  - 例: 自分のリソースかどうか、管理者ロールを持っているか等。
- バックエンド（API）側で必ず認可チェックを行い、フロント側の表示制御だけに頼らない。

---

## 2. 入力バリデーションとサニタイズ

### 2.1 入力バリデーション

- すべての外部入力（HTTP リクエスト、クエリパラメータ、ボディ、ヘッダ、環境変数など）に対して、**型・長さ・フォーマット** のバリデーションを行う。
- バリデーションはできるだけ **サーバー側でも** 実施し、フロント側のバリデーションにだけ頼らない。

### 2.2 XSS 対策

- フロントエンドで `dangerouslySetInnerHTML` や、同等の「生 HTML 挿入」を使う場合は **極力避ける**。どうしても必要な場合は、
  - 信頼できるソースだけに限定する、
  - サニタイズライブラリ（DOMPurify 等）を通す。
- テンプレートエンジン側（React 等）は基本的に XSS 対策済みなので、**ユーザー入力を文字列として埋め込むだけなら問題ない**。

### 2.3 SQL インジェクション対策

- 生の SQL を扱う場合は、**プレースホルダ付きのバインド**（`$1`, `?` など）を使い、文字列連結でクエリを組み立てない。
- ORM / クエリビルダを使う場合も、**動的に組み立てる部分（where 句など）に生の文字列を差し込まない**。

## 3. 機密情報の扱い

### 3.1 環境変数・設定

- API キー・DB パスワード・OAuth クライアントシークレットなどの機密情報は、
  - `.env` / 環境変数 / Secret Manager 等で管理し、
  - ソースコードや Git リポジトリには **直書きしない**。
- `.env` ファイルは `.gitignore` に追加し、公開リポジトリにコミットしない。
- サンプル用には `.env.sample` を用意し、**値は空やダミーにする**。

### 3.2 ログ

- ログに以下の情報を出力してはいけない：
  - パスワード
  - クレジットカード番号/セキュリティコード
  - アクセストークン / リフレッシュトークン
  - 個人情報（メールアドレスや住所など）は、必要最低限にとどめる。
- エラーログには内部エラーの詳細（スタックトレースなど）を残してもよいが、**クライアントへ返すレスポンスには詳細を出し過ぎない**（内部構造の漏えい防止）。

---

## 4. 通信・CORS

- 認証情報（Cookie / Authorization ヘッダ）を含む通信は、**本番環境では HTTPS 前提** とする。
- CORS 設定は **許可するオリジンを明示的に限定**する。
  - `Access-Control-Allow-Origin: *` としない。
  - 少なくとも `http://localhost:3000`（開発）と、実際の本番ドメインだけを許可する。

---

## 5. ファイルアップロード・外部入力（将来的な考慮）

- ファイルアップロードを実装する場合：
  - 許可する **拡張子 / MIME タイプ / サイズ** を明確に制限する。
  - アップロードファイルはアプリケーションサーバーとは分離されたストレージ（例: S3）に置く。
- 外部 URL をフェッチする機能を実装する場合：
  - SSRF（サーバーサイドリクエストフォージェリ）に注意し、**内部 IP (127.0.0.1, 169.254.169.254 等) や社内ネットワークへのアクセスをブロック**する。

---

## 6. デフォルト設定・その他

- ライブラリ・フレームワークの **セキュリティ関連のデフォルト設定**（CSRF 対策、ヘッダ追加など）は、**よほどの理由がない限り無効化しない**。
- エラー画面やスタックトレースを本番でそのままユーザーに見せない（「予期せぬエラーが発生しました」程度のメッセージに留める）。
- デバッグ用エンドポイントやツール（GraphQL Playground など）は、本番では認証付き or 無効化する。

---

## 7. AI への短い指示（コピペ用）

> このプロジェクトでは、`docs/SECURITY_GUIDE.md` に記載されたルールに従い、  
> - パスワードの平文保存・ログ出力を行わないこと  
> - SQL の文字列連結や危険な生 HTML 挿入を行わないこと  
> - 機密情報をソースコードに直書きしないこと  
> - 認証・認可チェックを必ずサーバー側で行うこと  
> を前提に実装してください。  
> セキュリティに影響しそうな実装が必要な場合は、このガイドの該当セクションを参照し、安全なパターンでコードを生成してください。
